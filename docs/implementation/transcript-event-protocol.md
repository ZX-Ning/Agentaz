# Transcript Event Protocol

## Status

Accepted for protocol v8.

## Context

Agentaz already streams assistant output through `message_upsert`,
`message_block_upsert`, and `message_block_delta`. The weak point was the
first user message path: the browser created an optimistic `local-*` message,
while the backend only made that user prompt visible later through HTTP
history. A forced history refresh from coarse `status` or `state_snapshot`
events could therefore temporarily replace the transcript with a payload that
did not yet contain the submitted user message.

The opencode investigation showed a useful boundary: user messages, assistant
parts, session status, and snapshots are separate event concerns. Agentaz does
not need to copy opencode's full message/part store, because `UiMessage` and
`UiBlock` already provide the app-level message/block projection we want.

## Decision

Protocol v8 keeps the existing `UiMessage`/`UiBlock` transcript model and adds
three explicit lifecycle concepts:

- `clientMessageId`: generated by the browser before prompt submit and echoed
  by the backend to reconcile optimistic user messages without text matching.
- `turnId`: generated by the backend for a prompt turn and attached to turn
  lifecycle events.
- `transcriptRevision`: a monotonic backend counter included in history and
  turn completion events so stale history responses can be ignored.

The backend emits `turn_started` before calling the Pi prompt loop. This event
contains the canonical user `UiMessage` that replaces the local optimistic
message. The backend emits `turn_completed` only after the prompt task
finishes, and that event is the frontend's normal signal to refresh HTTP
history. `turn_failed` reports failures while preserving the existing `error`
event for user-visible toasts; the frontend removes the matching local prompt
placeholder and refreshes history so preflight failures cannot leave orphaned
messages.

`follow_up` is intentionally outside this reconciliation path. It queues text
inside Pi's pending queue and does not send `clientMessageId` or emit
`turn_started` for a browser-optimistic user message. If follow-up becomes a
visible queued transcript item later, it should get a dedicated queue-message
protocol rather than borrowing prompt turn semantics.

`status` and `state_snapshot` no longer drive transcript refresh. They remain
responsible for session list, streaming flags, pending counts, approval state,
control ownership, and reconnect recovery.

## Consequences

- The first user message no longer depends on history refresh timing.
- Failed prompt preflight no longer leaves an optimistic/canonical orphan in
  the browser transcript.
- Transcript refresh is tied to an explicit completion boundary instead of a
  coarse idle/status inference.
- Existing assistant streaming stays unchanged.
- A future full message/part store remains possible, but it is not required for
  the current local-first Nuxt + Pi SDK architecture.
