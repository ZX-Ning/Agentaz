# Agentaz -- Code Review

BASED ON HEAD: 0e1f142cd9da8997814ca908a2c8b797096bd360

Reviewr: Claude Opus 4.6

## Executive Summary

Agentaz is a well-structured local-first Nuxt 4 app wrapping a Pi SDK coding agent with a browser UI. The architecture is clean: thin HTTP routes delegate through a process-wide singleton service graph (`AgentRuntime`), WebSocket is one-way (server-to-client events only), and the frontend uses a single-controller composable pattern. Documentation is excellent.

The codebase is in good shape for its stage but has several issues worth fixing, from a build-breaking dependency pin to missing resilience patterns and testing gaps.

---

## Architecture Overview

```
Browser (Vue SPA)
  │
  ├─ HTTP POST/PUT/GET ──▶ Nitro API Routes (thin) ──▶ agent-http helpers ──▶ AgentRuntime
  │                                                                              │
  └─ WebSocket (read-only) ◀── WsAgentHub ◀── AgentEventBus ◀──────────────────┤
                                                                                 │
                                                   ├── ClientPresence (focus/control leases)
                                                   ├── PiSessionWorkspace (session lifecycle)
                                                   │    └── PiSessionController[N] (per-session)
                                                   │         └── WebExtensionUIContext (UI bridge)
                                                   └── SessionProjector (state snapshots)
```

**Tech stack:** Nuxt 4 (SSR off), Nitro, Vue 3 + Nuxt UI 4, crossws WebSocket, `nuxt-auth-utils`, `@earendil-works/pi-coding-agent` SDK.

## Findings

### 1. Unpinned Pi SDK Dependency

- **Severity:** Critical
- **Location:** `apps/web/package.json:13` -- `"@earendil-works/pi-coding-agent": "latest"`
- **Problem:** The core SDK dependency uses `"latest"` instead of a pinned or range version. Any breaking change to the Pi SDK will silently break CI/CD and production builds on next `pnpm install`.
- **Impact:** Build failures, runtime crashes, or silent behavior changes with no way to reproduce the previous state.
- **Suggested fix:** Pin to a specific version (e.g., `"^1.2.3"`) or a semver range. Add a `pnpm-lock.yaml` audit step to CI.

### 2. No Graceful Shutdown

- **Severity:** High
- **Location:** `apps/web/server/plugins/startup.ts`, `apps/web/server/utils/agent-runtime.ts`
- **Problem:** There is no Nitro shutdown hook that calls `workspace.disposeAll()`. When the process terminates (Ctrl+C, deploy, crash), active Pi SDK sessions are not cleanly disposed. The `PiSessionWorkspace` has a `disposeAll()` method, but nothing calls it.
- **Impact:** Corrupted session files, leaked resources, orphaned subprocesses if the Pi SDK manages external processes.
- **Suggested fix:** Add a `close` hook to the Nitro plugin:
  ```ts
  nitroApp.hooks.hook('close', async () => {
    const { workspace } = getAgentRuntime();
    await workspace.disposeAll();
  });
  ```

### 3. Event Bus Handler Errors Crash Publishers

- **Severity:** High
- **Location:** `apps/web/server/utils/agent-event-bus.ts:28`
- **Problem:** `publish()` iterates handlers without try/catch. If any subscriber throws, the exception propagates to the publisher and prevents remaining subscribers from receiving the event.
- **Impact:** A bug in one subscriber (e.g., WsAgentHub) could crash the PiSessionWorkspace or PiSessionController's event emission, breaking the agent loop.
- **Suggested fix:** Wrap each handler call in try/catch with `console.error`:
  ```ts
  for (const handler of this.handlers) {
    try { handler(event); } catch (e) { console.error('[event-bus]', e); }
  }
  ```

### 4. No WebSocket Reconnection

- **Severity:** High
- **Location:** `apps/web/app/composables/useAgentaz.ts` -- `connectWebSocket()`
- **Problem:** If the WebSocket disconnects (server restart, network blip), the status changes to `"disconnected"` but there is no automatic reconnect. The user must reload the entire page.
- **Impact:** Poor reliability during development (server restarts frequently) and any network instability. All realtime state updates stop.
- **Suggested fix:** Implement exponential-backoff reconnection with jitter. On reconnect, re-fetch state and history for the active session.

### 5. WebSocket `peer.send()` Not Guarded

- **Severity:** High
- **Location:** `apps/web/server/utils/ws-agent-hub.ts:214`
- **Problem:** `broadcast()` calls `peer.send()` for every peer without try/catch. If one peer's send fails (disconnected but not yet cleaned up), the exception stops delivery to all remaining peers.
- **Impact:** One broken WebSocket connection could black-hole events for all connected browsers.
- **Suggested fix:** Wrap each `peer.send()` in try/catch:
  ```ts
  private broadcast(event: ServerEvent) {
    const data = JSON.stringify(event);
    for (const peer of this.peers.values()) {
      try { peer.send(data); } catch (e) { console.error('[ws-hub] send failed', e); }
    }
  }
  ```
  Also: serialize once, send the same string to all peers (minor perf win already shown above).

### 6. Missing Test Infrastructure

- **Severity:** High
- **Location:** No `vitest.config.ts`, no test runner in dependencies, `scripts/smoke-backend.mjs` covers ~30% of endpoints.
- **Problem:** The only test is a smoke script that tests login, state, session create, history, models, and thinking. It does **not** test: message submission, abort, queue clear, model set, session focus, UI request response, or WebSocket event flow. There are no unit tests for any utility, composable, or component.
- **Impact:** Regressions in core features (prompt submission, approval flow, session eviction) are not caught before deploy.
- **Suggested fix:**
  1. Add Vitest as a dev dependency with a config file.
  2. Unit test pure utilities: `agentaz-session-list.ts`, `agentaz-transcript.ts`, `app.util.ts`, `client-presence.ts`, `session-projector.ts`.
  3. Expand the smoke test to cover message submission, abort, and UI request endpoints.
  4. Add Vue component tests for `ChatMessage.vue` (complex rendering logic).

### 7. `ui_input_request` Has No Text Input

- **Severity:** Medium
- **Location:** `apps/web/app/components/PendingUiRequests.vue:68-75`
- **Problem:** The `ui_input_request` type includes a `placeholder` field, but the component only offers "Submit empty" and "Cancel" buttons. There is no actual text input field for the user to type a response.
- **Impact:** Users cannot provide text input to extensions that request it. This silently degrades extension functionality.
- **Suggested fix:** Add a `<UInput>` bound to a local ref, with the extension's `placeholder`, and a submit button that emits the typed value.

### 8. Substring-Based Error Classification

- **Severity:** Medium
- **Location:** `apps/web/server/utils/agent-http.ts` -- `agentHttpError()`
- **Problem:** HTTP error codes are assigned by checking `error.message` substrings like `"No loaded session"`, `"controlled by another browser client"`, etc. If the Pi SDK or workspace changes error wording, the classification breaks silently, returning 500 instead of the correct 4xx.
- **Impact:** Frontend error handling depends on status codes; wrong codes produce confusing UX.
- **Suggested fix:** Define typed error classes or error codes in the workspace/controller layers. Classify by error type/code, not message string.

### 9. Optimistic Send Without Failure Recovery

- **Severity:** Medium
- **Location:** `apps/web/app/composables/useAgentaz.ts:518-523`
- **Problem:** When the user sends a prompt, a local user message is immediately pushed to the transcript. If the backend POST fails, the message remains in the transcript with no failure indication.
- **Impact:** The user sees their message but gets no feedback that it wasn't actually sent. The ghost message persists until page reload.
- **Suggested fix:** Add a `status: 'sending' | 'sent' | 'failed'` field to user messages. Mark as `failed` in the catch block with a retry affordance.

### 10. `runtimeConfig` Uses `||` Instead of `??`

- **Severity:** Medium
- **Location:** `apps/web/nuxt.config.ts:32-34`
- **Problem:** `Number(process.env.PI_WEB_APPROVAL_TIMEOUT_MS || 5 * 60 * 1000)` uses `||`. If the env var is set to `"0"` (falsy string), the fallback kicks in instead of respecting the explicit zero. Same pattern for `maxLoadedSessions`.
- **Impact:** Cannot set these values to `0` via env vars. Low practical impact since `0` is not a sensible value for either, but it is still a latent bug pattern.
- **Suggested fix:** Use `??` instead of `||`, or use a helper that distinguishes between undefined/empty and falsy.

### 11. Health Endpoint Requires Authentication

- **Severity:** Medium
- **Location:** `apps/web/server/middleware/auth.ts:4` -- `PUBLIC_API_PATHS` does not include `/api/health`
- **Problem:** `GET /api/health` is behind the auth middleware. External monitoring tools, load balancers, and container orchestration health checks typically do not carry session cookies.
- **Impact:** Cannot use standard health check probes without authentication.
- **Suggested fix:** Add `/api/health` to `PUBLIC_API_PATHS`.

### 12. Transcript Memory Growth Is Unbounded

- **Severity:** Medium
- **Location:** `apps/web/server/utils/pi-session-controller.ts:104` -- `transcript` Map
- **Problem:** The `transcript` Map grows unboundedly during a session's lifetime. Tool blocks are cleared on `agent_end`, but messages accumulate forever. A very long coding session could consume significant memory.
- **Impact:** Memory pressure for sessions with hundreds of turns. Mitigated by `maxLoadedSessions` (default 5) and session eviction, but still a concern for long-running sessions.
- **Suggested fix:** Consider pruning older messages from the in-memory transcript after a threshold, keeping them only in the persisted session file.

### 13. `declare function useColorMode()` Workaround

- **Severity:** Low
- **Location:** `apps/web/app/components/AppHeader.vue:4-7`, `apps/web/app/pages/login.vue`
- **Problem:** Both files manually declare `useColorMode()` to work around missing auto-import types. This is fragile -- if `@nuxtjs/color-mode` changes its return type, the declared type silently diverges.
- **Impact:** Potential type mismatches masked at compile time.
- **Suggested fix:** Add proper type augmentation in a `types/color-mode.d.ts` file, or configure `@nuxtjs/color-mode` module types correctly in `tsconfig.json`.

### 14. `ChatMessage.vue` Mutates State During Render

- **Severity:** Low
- **Location:** `apps/web/app/components/ChatMessage.vue:121-124` -- `isBlockCollapsed()`
- **Problem:** `isBlockCollapsed()` lazily writes default values to `collapsedStates` during the render cycle. This is a Vue anti-pattern that can trigger cascading re-renders.
- **Impact:** Minor performance overhead; no correctness bug. Could cause unexpected behavior if Vue warns about state mutation during render in future versions.
- **Suggested fix:** Use a computed map that provides defaults without mutation, or initialize `collapsedStates` in a watcher on `renderedBlocks`.

### 15. Event Bus Subscription Never Cleaned Up

- **Severity:** Low
- **Location:** `apps/web/server/utils/ws-agent-hub.ts:144-148`
- **Problem:** `startEventSubscription()` subscribes to the `AgentEventBus` once and never unsubscribes, even when all WebSocket clients disconnect. The handler fires on every runtime event, iterating an empty peers map.
- **Impact:** Negligible CPU cost (a few function calls per event). No memory leak since the handler is a single closure. But it is conceptually untidy.
- **Suggested fix:** Unsubscribe in `stopHeartbeat()` when peers reach zero, and clear `this.unsubscribe`.

### 16. Stale Documentation

- **Severity:** Low
- **Location:** `docs/roadmap.md:82`
- **Problem:** States "The current app has no auth" -- this is factually incorrect. Auth (admin password + encrypted cookie) is fully implemented.
- **Impact:** Misleads contributors about the security posture.
- **Suggested fix:** Update the roadmap to reflect that auth exists and note remaining security work (CSRF, origin checks).

### 17. No CSRF or Origin Validation

- **Severity:** Low (local-first context)
- **Location:** `apps/web/server/middleware/auth.ts`, `apps/web/server/routes/api/agent/ws.ts`
- **Problem:** No CSRF token validation on state-mutating POST/PUT requests. No `Origin`/`Referer` header check on WebSocket upgrade. In a local-first context with `127.0.0.1` binding, this is mitigated but not eliminated -- a malicious webpage opened in the same browser could make cross-origin requests to `localhost:3000`.
- **Impact:** Low for the target deployment (local single-user), but a prerequisite if the app is ever exposed beyond localhost.
- **Suggested fix:** Already noted in `docs/roadmap.md` as future work. Add `SameSite=Strict` to the session cookie. Consider a simple Origin check on the WebSocket upgrade handler.

### 18. `x-agentaz-client-id` Header Is Not Validated

- **Severity:** Low (local-first context)
- **Location:** `apps/web/server/utils/agent-http.ts` -- `requestClientId()`
- **Problem:** The `x-agentaz-client-id` header is trusted as-is. A client could spoof another tab's client ID to steal control leases.
- **Impact:** In a single-user local app, the user would be impersonating themselves. No practical impact currently, but a design gap if multi-user is ever considered.
- **Suggested fix:** Validate that the client ID matches the authenticated session, or bind it to the WebSocket connection ID.

## Positive Observations

The codebase deserves credit for several strong patterns:

- **Clean layering:** Routes are thin, services are separated by responsibility, the event bus decouples producers from consumers.
- **Re-entrant control leases** in `ClientPresence` are a sophisticated solution for the fire-and-forget message pattern.
- **Deferred settings** (model/thinking changes while streaming) are handled correctly with snapshot-and-clear atomicity.
- **Lazy initialization with promise dedup** in `PiSessionController.ensureInitialized()` handles concurrent callers correctly.
- **Fail-closed auth** -- the server refuses to start without proper secrets configured.
- **Timing-safe password comparison** prevents timing attacks.
- **Excellent documentation** -- 8 docs files with Mermaid diagrams, performance measurements, and debugging playbooks.
- **`content-visibility: auto`** on chat messages is a smart browser-native performance optimization.
- **Conservative default permissions** in the permission config (mutating tools require approval, `.env` denied).
